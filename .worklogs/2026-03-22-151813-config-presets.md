# Curated English config presets

**Date:** 2026-03-22 15:18
**Scope:** `apps/prosesmasher/presets/*`, `apps/prosesmasher/crates/adapters/outbound/fs/src/config_loader_tests.rs`

## Summary
Added a curated preset library under `apps/prosesmasher/presets/` and wired it into the automated test surface by loading each preset through `FsConfigLoader`. The presets provide English-first starting points for general prose, blogs, technical articles, docs, landing pages, and essays.

## Context & Problem
After the rewrite-oriented JSON/evidence work, the main remaining product gap was usability. The tool could validate prose well, but users still had to invent a full config from scratch before they could use it in a generation loop. That is too much setup friction for a tool that is supposed to sit directly in the writing/rewrite path.

The user asked for “good config presets for everything.” The practical interpretation was to ship a meaningful preset library rather than wait on a separate CLI preset-resolution feature. The config loader already accepts JSON files cleanly, so preset files are immediately usable with `--config path`.

## Decisions Made

### Ship preset files first, not CLI preset-name resolution
- **Chose:** Add curated JSON files under `apps/prosesmasher/presets/` and validate them through tests.
- **Why:** This delivers value immediately with no CLI surface expansion or resolution logic. Users can already point `--config` at a preset file.
- **Alternatives considered:**
  - Add a new `--preset` flag first — rejected because preset-name resolution is a second step and not necessary to prove the preset library itself.
  - Wait for `prosesmasher init` before creating presets — rejected because the preset content is useful independently of future CLI UX.

### Make the presets English-first
- **Chose:** Create English presets only.
- **Why:** Many phrase and rhetoric checks are explicitly English in tone and structure. Shipping fake multilingual presets without real curation would reduce quality and trust.
- **Alternatives considered:**
  - Generate presets for all supported locales using translated lists — rejected because that would be low-confidence and likely wrong.
  - Restrict presets to structure/readability only for all locales — rejected because the user asked for useful presets, not intentionally incomplete ones.

### Organize presets by content type, not by arbitrary strictness ladder alone
- **Chose:** Add `general-en`, `blog-strict-en`, `technical-article-en`, `docs-en`, `landing-page-en`, and `essay-en`.
- **Why:** Content type changes the right thresholds materially. A landing page and a technical article should not share word-count or readability targets even if they share the same anti-slop term lists.
- **Alternatives considered:**
  - Only `strict/standard/lenient` presets — rejected because that hides the more meaningful distinction: content type.
  - One universal preset — rejected because it would be mediocre everywhere.

## Architectural Notes
The presets live in the app workspace rather than inside adapter test fixtures because they are intended as product artifacts, not just test data. The tests in `config_loader_tests.rs` load them through the same `FsConfigLoader` used in production, which means:
- parsing stays honest
- DTO conversion stays covered
- future schema changes will break tests if presets drift

The preset README is intentionally short and practical. It frames the files as starting points, not canonical truth, because domain-specific terms and required terminology still belong in downstream customized configs.

## Open Questions / Future Considerations
- The CLI still lacks preset name resolution. Right now users must pass a path explicitly.
- Presets are English-only; multilingual preset strategy remains open.
- Required/recommended terms are intentionally empty in the presets because those should usually be project-specific. We may eventually want “template presets” with placeholder sections for those lists.

## Next Steps / Continuation Plan
1. Add CLI support for preset selection by name, likely via `--preset <name>` or future default-config discovery.
2. Decide whether presets should be copied into user projects (`init`) or referenced in place.
3. Add one or two example commands in project docs showing how to use a preset in an LLM rewrite loop.
4. If multilingual adoption matters, design a separate preset strategy for locale-neutral structure/readability profiles versus language-specific term/rhetoric profiles.
